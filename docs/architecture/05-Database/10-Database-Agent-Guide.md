# Non-Negotiable Rules

## 1. Database is source of truth

- Store facts. Derive statistics in views only.
- Never persist averages, percentages, or other derivable values without measured justification.
- Completed gameplay is immutable. Corrections = new records, never UPDATE on historical rows.
- This immutability is enforced by the application layer; the docs do not assume database triggers for workflow enforcement.

## 2. Hybrid identifiers

| Entity type | PK | Generated by |
| ----------- | -- | ------------ |
| Domain entities | UUIDv7 | Application |
| Lookup tables | SMALLINT | Seed scripts (fixed ids) |

Every INSERT supplies `id`. The database never generates identifiers.

Domain entities: players, activities, exercise_sessions, turns, darts, templates, game_types, ruleset_versions, configuration_templates.

Lookups: statuses, dart_zones, participant_types, stage_types, capture_modes, input_modes, duration_types, game_features.

## 3. Template → Snapshot → Session

```
configuration_templates (preset, mutable)
    ↓ copy at session start
exercise_configurations (JSONB snapshot, immutable)
    ↓
exercise_sessions (runtime)
```

Runtime tables **never** reference template tables. No template FK on sessions.

## 4. Dart facts

One row = one thrown dart. Store:

- `intended_target_number` + `intended_zone_id`
- `hit_target_number` + `hit_zone_id`
- `score`

No `multiplier` column (derived from zone). Recreational capture may omit dart rows entirely (turn totals only).

## 5. Views are API contracts

- Prefix: `v_*` (not `vw_*`)
- API reads views, never raw runtime tables
- Expose `implementation_key`s (as `*_key`), not internal lookup ids; keep only entity UUIDs (see `01-Naming-Conventions.md` §"View Column Key And Label Naming")
- Six implemented views (`0009` originals normalized by `0013`/`0014`; `0016` adds `v_configuration_presets` and rebuilds replay/overview); new views = new migration

## 6. Migrations

- One responsibility per file
- Four-digit prefix: `0001`–`0016` current chain
- Schema in migrations; reference data in seeds
- **Never modify an applied migration** — create a migration with the next unused number instead
- Before first deployment only: in-place correction of unapplied migrations is permitted
- No `BEGIN`/`COMMIT` in migrations (dbmate wraps each section in a transaction); seeds keep explicit `BEGIN`/`COMMIT`

## 7. Controlled sets = lookup tables

Never use free-text where a reference table exists. Example: `duration_type_id SMALLINT` FK to `duration_types`, not `duration_type TEXT`.

## 8. Ruleset-owned limits

Max darts per turn, score caps, bust rules — enforced by the application from the ruleset version. Not DB CHECK constraints.

## 9. Ordering uniqueness

Enforced in `0011`:

- `uq_routine_steps_sequence`
- `uq_turns_stage_participant_sequence`
- `uq_darts_turn_number`
- Stage ordering via partial unique indexes (nullable parent)
- `uq_sessions_single_active` — one active session per player per game type

## 10. JSONB configuration

Both `configuration_templates.configuration` and `exercise_configurations.configuration` are JSONB objects. Structure per game type is defined by the ruleset version. Application validates; DB only CHECKs `jsonb_typeof = 'object'`.

# Before You Change the Database

Answer these questions:

1. **Which layer?** Reference / Player / Template / Runtime / Read Model
2. **What is the owner?** Every entity has exactly one owner
3. **Is it a fact or a derivation?** Facts go in tables; derivations go in views
4. **Does runtime reference a template?** If yes, redesign — copy instead
5. **Is this a new migration or a seed?** Schema → migration. Static data → seed
6. **Does a view or API contract break?** Update views in a dedicated migration
7. **Does documentation need updating?** Update `06-Database-Specification.md` and `03-Migrations.md`

# Migration Checklist

```
[ ] One clear responsibility
[ ] Numbered sequentially after the current chain end (0016)
[ ] No BEGIN/COMMIT (dbmate owns the transaction)
[ ] FK constraints with fk_ prefix
[ ] CHECK constraints with chk_ prefix
[ ] Unique constraints with uq_ prefix
[ ] Indexes only with documented query pattern (idx_ prefix)
[ ] No seed data in migration file
[ ] 06-Database-Specification.md updated if entities change
[ ] 03-Migrations.md updated if chain changes
```

---

# Seed Checklist

```
[ ] Fixed explicit ids (SMALLINT for lookups, UUID ranges for domain)
[ ] ON CONFLICT DO NOTHING for idempotency
[ ] System templates: is_system_template = TRUE, player_id IS NULL
[ ] No user data, no runtime history, no test data
```

Current seeds:

- `0001_reference_data.sql` — lookups + game types + rulesets + dart zones
- `0002_default_templates.sql` — exercise templates, configuration presets, one system routine

# Forbidden Actions

| Action | Why |
| ------ | --- |
| Modify applied migration files | Breaks reproducibility |
| Expose raw tables to API | Violates CQRS-lite |
| Store derivable statistics | Violates Facts Over Calculations |
| Add template FK to runtime | Breaks historical accuracy |
| Use DB-generated UUIDs | Violates identifier strategy |
| Add multiplier column to darts | Duplicates zone truth |
| Create EAV / polymorphic FKs for gameplay | Violates Explicit Domain Modeling |
| Index every FK blindly | Write overhead without query evidence |
| Put business logic in views beyond joins/labels | Views are read contracts, not workflow engines |

# Safe Change Patterns

## Add a new game type

1. Seed row in `game_types` (UUID + implementation_key)
2. Seed feature mappings in `game_type_features`
3. Seed `ruleset_versions` row
4. Seed `configuration_templates` presets in `0002` or new seed
5. Seed `exercise_templates` if needed
6. Frontend game engine (outside database)
7. Future: game-specific analytics view

No existing table changes required.

## Add a column to runtime table

1. New migration with the next unused number (currently `0017_add_<column>.sql`)
2. Update `06-Database-Specification.md` entity section
3. Update views if API needs the column
4. Update `03-Migrations.md` chain

## Add a new view

1. New migration (or extend `0009` only if not yet applied anywhere)
2. Document in `05-Views.md` and `06-Database-Specification.md` Read Model Layer
3. API repository reads the view

## Add a new lookup value

1. Add row to appropriate seed file with next available SMALLINT id
2. Document in `06-Database-Specification.md` entity section
3. No migration needed if table already exists

# When Uncertain

1. Read `06-Database-Specification.md` for the entity in question
2. Read the matching migration file for actual column names
3. Check `04-Architecture-patterns.md` for the applicable pattern
4. If the change affects architecture, stop and propose a design before implementing
5. Never guess at JSONB configuration key vocabulary — check `seeds/0002_default_templates.sql` and ruleset version
