# Database Migration Strategy

> **Version:** 1.5.0
>
> This document defines the migration strategy and operating principles for evolving the PostgreSQL database.
>
> Database changes must be predictable, reviewable and reproducible throughout the lifetime of the application.

---

# Purpose

Migrations are the controlled mechanism for changing the database structure.

They ensure:

- consistent environments
- reproducible database creation
- traceable schema evolution
- safe collaboration
- predictable deployments

The database must always be rebuildable from:

```
Empty database

↓

Migrations

↓

Seeds

↓

Ready application database
```

---

# Migration Principles

Every migration must follow these principles:

- one responsibility per migration
- deterministic execution
- explicit ordering
- reviewable changes
- no hidden side effects

A migration should answer:

> "What database capability does this migration introduce?"

---

# Migration Ownership

Database migrations are the source of truth for:

- tables
- columns
- constraints
- indexes
- views
- database functions
- extensions

Migrations are not the source of truth for:

- runtime data
- user data
- temporary data

# dbmate Format

Migration files executed by `dbmate` must use section markers:

```sql
-- migrate:up
-- DDL statements (no explicit BEGIN/COMMIT)

-- migrate:down
-- structural reverse DDL
```

Rules:

- Every migration requires both `-- migrate:up` and `-- migrate:down`.
- Down blocks reverse structural DDL only (DROP TABLE/VIEW/INDEX/CONSTRAINT).
- Do not wrap sections in `BEGIN`/`COMMIT` — dbmate runs each section in a transaction.
- Header comments and inline documentation are preserved above the markers.

Example:

```sql
-- ============================================================
-- Migration: 0001_extensions.sql
-- ============================================================

-- migrate:up
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- migrate:down
DROP EXTENSION IF EXISTS pg_stat_statements;
```

---

# Migration Naming

Migration files use:

```
<number>_<description>.sql
```

Numbers use a four-digit zero-padded prefix.

Examples:

```
0001_extensions.sql

0006_runtime_events.sql

0011_ordering_and_uniqueness.sql
```

Rules:

- numbers are sequential
- names describe responsibility
- files are never renamed after execution
- previous migrations are never rewritten after execution

Before first deployment, defects in unapplied migrations may be corrected in place. After a migration has been applied to any shared environment, corrections require a new migration.

# Migration Execution Order

Migrations execute in dependency order.

Current order:

---

## 0001_extensions.sql

Purpose:

Enable required PostgreSQL extensions.

Contains:

- pg_stat_statements

UUIDs are generated by the application, so no UUID extension is required.

---

## 0002_reference_tables.sql

Purpose:

Create controlled application definitions.

Contains:

- game types
- game features and mappings
- statuses
- capture modes
- input modes
- duration types
- participant types
- stage types
- ruleset versions
- dart zones

Lookup tables use SMALLINT identifiers.

Domain-level reference entities (game types, ruleset versions) use UUIDv7.

Reference data itself is inserted through seeds.

---

## 0003_players.sql

Purpose:

Create player-related structures.

Contains:

- players (application profile)
- player_settings (shared primary key)

Authentication remains external.

The database does not store authentication credentials.

---

## 0004_templates.sql

Purpose:

Create reusable definitions.

Contains:

- exercise_templates
- routine_templates
- routine_steps (duration through duration_type_id foreign key)

Templates describe possible future gameplay.

They do not represent historical execution.

---

## 0005_runtime_core.sql

Purpose:

Create runtime session structures.

Contains:

- activities
- exercise_sessions
- exercise_configurations (JSONB snapshot)
- participants
- exercise_stages

This defines active and historical gameplay containers.

---

## 0006_runtime_events.sql

Purpose:

Create gameplay event storage.

Contains:

- turns
- darts (intention + result model)

This is the foundation for:

- replay
- analytics
- progression tracking

The dart is the smallest gameplay event.

---

## 0007_constraints.sql

Purpose:

Add integrity rules.

Contains:

- check constraints
- unique constraints
- domain validation

Constraints ensure invalid states cannot be stored.

Ruleset-owned limits (maximum darts per turn, score caps) are deliberately not enforced here.

---

## 0008_indexes.sql

Purpose:

Add performance optimization.

Indexes are based on:

- query patterns
- expected workload
- measured performance

Indexes should not be created automatically for every column.

---

## 0009_views.sql

Purpose:

Create application-facing read models.

Views provide stable query contracts for the API.

Contains:

- v_active_sessions
- v_session_overview
- v_game_replay
- v_dart_analytics
- v_routine_execution

---

## 0010_configuration_templates.sql

Purpose:

Create reusable configuration presets.

Completes the configuration chain:

```
Template → Snapshot → Runtime Session
```

Contains:

- configuration_templates (JSONB presets, system and user owned)

---

## 0011_ordering_and_uniqueness.sql

Purpose:

Enforce event ordering integrity and lifecycle rules.

Contains:

- unique routine step ordering
- unique stage ordering (partial indexes for root and sibling stages)
- unique turn ordering per participant per stage
- unique dart numbering per turn
- single active session per player per game type (unique partial index)
- removal of 0008 indexes made redundant by the new unique constraints

---

## 0012_session_write_idempotency.sql

Purpose:
Document the schema artifact that backs idempotent batch writes.

Contains:

- session_write_idempotency

Stores persisted outcomes for `POST /api/sessions/:sessionId/events/batch`.

Uniqueness:

- same session + same idempotency key

---

## 0013_normalize_read_model_views.sql

Purpose:

Normalize read-model view columns to one consistent standard.

Contains:

- rewritten v_active_sessions, v_session_overview, v_game_replay, v_dart_analytics, v_routine_execution

Renames implementation-key/label columns to `<concept>_key` / `<concept>_name`, drops internal lookup ids, and adds `ruleset_version_key` to `v_active_sessions`. Behaviour-preserving; never edits `0009`.

---

## 0014_dart_analytics_session_scope.sql

Purpose:

Session-scope the dart analytics read model.

Contains:

- rewritten v_dart_analytics (adds session_id)

Adds `session_id` so `GET /api/sessions/:sessionId/darts` can filter by session through the view; `player_id` retained for future player-global statistics. Behaviour-preserving; never edits `0013`.

---

## 0015_time_semantics_constraints.sql

Purpose:

Align time constraints with the batch write model. <!-- 2026-07-13 -->

Contains:

- drop `chk_turn_completed_after_created` (client-observed `turns.completed_at` legitimately predates row insert under batch upload)
- replace `chk_players_display_name_not_empty` without the dead NULL arm

Doctrine: `created_at` is row persistence time (server clock); gameplay chronology comes from sequence numbers and client-observed timestamps.

---

## 0016_read_model_replay_and_presets.sql

Purpose:

Rebuild replay/overview read models; add the preset read model. <!-- 2026-07-13 -->

Contains:

- rewritten `v_game_replay` (LEFT JOIN darts, `turn_total_score`, `stage_id`/`parent_stage_id` for tree reconstruction)
- rewritten `v_session_overview` (`duration_seconds` floored to integer)
- new `v_configuration_presets` backing `GET /api/configuration-templates`

Never edits `0009`/`0013`/`0014`.

# Schema Changes


Schema changes must be introduced through migrations.

Examples:

Adding a column:

```
new migration
↓
ALTER TABLE
↓
constraint update
↓
index update if required
```

Never manually modify production databases.

---

# Migration Atomicity

dbmate wraps each `-- migrate:up` and `-- migrate:down` section in a transaction automatically.

Authors should **not** add explicit `BEGIN`/`COMMIT` inside migration files.

PostgreSQL supports transactional DDL, so a failed migration rolls back cleanly.

If a migration fails:

- changes should rollback
- database should remain consistent

---

# Migration Size

Migrations should be small and focused.

Good:

```
0011_ordering_and_uniqueness.sql
```

Avoid:

```
0011_complete_database_restructure.sql
```

Large migrations are difficult to:

- review
- debug
- rollback

---

# Changing Existing Tables

When modifying an existing table:

Do not:

- edit the original migration after execution
- rewrite history
- remove existing columns without migration planning

Instead:

Create a new migration.

Example:

Existing:

```
0006_runtime_events.sql
```

New change:

```
0017_add_dart_coordinates.sql
```

---

# Data Migrations

Schema migrations and data migrations should be separated when possible.

Avoid:

```
ALTER TABLE players ADD COLUMN nickname;

UPDATE every player record;
```

inside one migration unless tightly coupled.

Prefer:

```
0017_add_nickname_column.sql

0018_migrate_existing_nicknames.sql
```

---

# Reference Data Management

Reference data is managed through seeds.

Examples:

- game statuses
- dart zones
- participant types

Location:

```
database/seeds/
```

Reference seeds must be:

- deterministic
- idempotent
- environment independent

Example:

```sql
INSERT INTO dart_zones (...)
VALUES (...)
ON CONFLICT DO NOTHING;
```

---

# Seed Files

## 0001_reference_data.sql

Controlled lookup values and domain reference entities:

- game statuses, capture modes, input modes, duration types
- participant types, stage types
- game features and game types with feature mappings
- initial ruleset versions
- dart zones

## 0002_default_templates.sql

System-owned default templates:

- default exercise templates (one per game type)
- default configuration presets per game type
- one pre-configured training routine with ordered steps

All seed rows supply explicit fixed identifiers.

---

# What Belongs in Seeds

Allowed:

- static definitions
- lookup values
- system configuration
- system-owned templates

Examples:

```
DOUBLE

TREBLE

ACTIVE

501 — Best of 5 Legs
```

---

# What Does Not Belong in Seeds

Never store:

- users
- games
- sessions
- test data
- runtime history

---

# Migration Testing

Before applying migrations:

Validate:

- clean database creation
- migration order
- seed execution
- constraints
- indexes
- views

The complete flow should work:

```
CREATE DATABASE

↓

RUN ALL MIGRATIONS

↓

RUN SEEDS

↓

START APPLICATION
```

---

# Migration Execution (Application Workflow)

Migrations are executed from `app/` using `dbmate` with SQL stored under `database/migrations/`.

Required environment settings:

```
DATABASE_URL=<pooled connection string>
DBMATE_MIGRATIONS_DIR=../database/migrations
DBMATE_SCHEMA_FILE=../database/schema.sql
```

Standard validation: `npm run validate:app` (sole definition: `app/CLAUDE.md`). Seeding is a provisioning step, not validation — `npm run db:seed` runs when creating or resetting an environment (see `11-Neon-Integration.md`). <!-- 2026-07-14 -->

# Migration Review Checklist

Every migration review should verify:

## Purpose

- Does the migration have one clear responsibility?

## Ownership

- Does the new data have a clear owner?

## Integrity

- Are constraints required?

## Performance

- Are indexes justified?

## API Impact

- Do views or contracts require changes?

## Historical Impact

- Could existing data become invalid?

---

# AI Agent Migration Rules

AI agents modifying the database must:

1. Inspect existing migrations before creating changes.
2. Never modify executed migrations.
3. Create new numbered migrations.
4. Follow naming conventions.
5. Separate schema and seed changes.
6. Add constraints deliberately.
7. Add indexes only with query justification.
8. Update related documentation.

---

# Migration Anti-Patterns

Avoid:

## Editing Previous Migrations

Bad:

```
Modify 0005_runtime_core.sql after deployment
```

Good:

```
Create 0017_runtime_update.sql
```

---

## Mixing Responsibilities

Bad:

```
Create tables

+

Insert users

+

Create indexes
```

Good:

Separate responsibilities.

---

## Manual Database Changes

Bad:

```
Direct production SQL changes
```

Good:

```
Migration

↓

Review

↓

Deployment
```

---

# Final Principle

Migrations are the historical record of database evolution.

They must be treated as permanent architectural documentation.

A future developer or AI agent should be able to understand how the database evolved by reading the migration history alone.
