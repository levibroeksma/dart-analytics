# DB Read-Model Normalization (Design)

> **Status:** Design (pre-freeze)
> **Date:** 2026-07-12
> **Branch:** `response-shape-decisions`
> **Cycle:** 1 of 2. This cycle normalizes and re-freezes the database read-model **views**. Cycle 2 (separate spec) defines the camelCase API response DTOs on top of the clean views and freezes `06-API/03`+`04`.

---

## 1. Purpose

The v1 freeze validation focused on the API contract and did not validate the **read-model views** (`0009_views.sql`) for internal consistency. Inspection of the applied SQL and the introspected `app/src/db/schema.ts` shows the five views use **inconsistent column conventions**, **leak internal lookup ids**, and have **one functional gap**. This cycle fixes the read model at the source so the API response contracts (cycle 2) and the frontend can build on a robust, consistent foundation.

This is a database read-model change: a new migration `0013`, re-introspection, and the cascading documentation/context updates. It does **not** define API DTOs or implement any read endpoint — those are cycle 2 and later.

### Motivation (what the freeze missed)

Same concept, different names across views:

| Concept (implementation key) | `v_active_sessions` | `v_session_overview` | `v_dart_analytics` |
| ---- | ---- | ---- | ---- |
| game type | `game_type_key` | `game_type` | `game_type` |
| capture mode | `capture_mode_key` | `capture_mode` | — |

Plus: views expose raw internal ids (`game_type_id`, `capture_mode_id`, `input_mode_id`, `ruleset_version_id`) that the API contract should never surface (the contract is key-based — clients send and receive `implementation_key`s, never internal ids); replay/analytics use bare `intended_zone` / `hit_zone` / `participant`; and `v_active_sessions` exposes `ruleset_version_id` (a UUID) but **no ruleset key**, so a resuming client cannot name the ruleset without a second lookup.

### TypeScript convention (the robust target, already partly in place)

The stack already implies the standard convention; the mess is semantic, not casing:

- **Postgres → snake_case** (idiomatic SQL) — unchanged.
- **TypeScript → camelCase**, mapped from snake_case by Drizzle at the DB boundary (`text("implementation_key")` → `implementationKey`) — already generated by `drizzle-kit introspect`.
- **JSON API responses → camelCase** (idiomatic JSON; matches the camelCase request contracts) — cycle 2.
- **Contract source of truth → Zod** response schema per endpoint, `type = z.infer<…>` — cycle 2, matching the frozen `03` convention.
- **Mapping seam → the repository layer** (one typed seam; snake_case never leaks past it) — cycle 2.

Cycle 1 makes the **view column names** (the layer Drizzle mirrors into TS) consistent so the camelCase names the whole stack inherits are consistent too.

## 2. The view-column naming standard (frozen by this cycle)

Recorded canonically in `05-Database/01-Naming-Conventions.md`:

1. **Implementation keys →** `<concept>_key` (e.g. `game_type_key`, `capture_mode_key`, `input_mode_key`, `status_key`, `stage_type_key`, `intended_zone_key`, `hit_zone_key`, `duration_type_key`, `ruleset_version_key`).
2. **Human labels →** `<concept>_name`, exposed **only** where a screen renders the label directly: `game_type_name`, plus the existing entity names `routine_name` / `exercise_name`. Enum-like lookups (capture/input mode, status, dart zones, stage types) are **key-only**; the frontend owns their presentation.
3. **No internal lookup ids in read models.** Drop `game_type_id`, `capture_mode_id`, `input_mode_id`, `ruleset_version_id`, and every other lookup `*_id`. Keep only entity UUIDs a client must address in a later request: `session_id`, `routine_id`, `exercise_template_id`, and `player_id` (retained for player-scoping filters).
4. **Behavior is unchanged.** Only column names and projection change — same rows, same joins, same `WHERE` clauses. No new truth, no business logic (read-model layer invariant preserved).

## 3. Normalized views — migration `0013_normalize_read_model_views.sql`

A new migration. It never edits the applied `0009`. Because columns are renamed and dropped (not just appended), each view is `DROP VIEW` then `CREATE VIEW` (not `CREATE OR REPLACE`). The `migrate:down` recreates the exact `0009` definitions.

### `v_active_sessions` (adds `ruleset_version_key`; drops 4 ids)

```sql
CREATE VIEW v_active_sessions AS
SELECT es.id AS session_id,
    es.player_id,
    gt.implementation_key AS game_type_key,
    gt.name             AS game_type_name,
    cm.implementation_key AS capture_mode_key,
    im.implementation_key AS input_mode_key,
    rv.implementation_key AS ruleset_version_key,
    es.started_at
FROM exercise_sessions es
    JOIN game_types gt      ON gt.id = es.game_type_id
    JOIN capture_modes cm   ON cm.id = es.capture_mode_id
    JOIN input_modes im     ON im.id = es.input_mode_id
    JOIN ruleset_versions rv ON rv.id = es.ruleset_version_id
    JOIN game_statuses gs   ON gs.id = es.status_id
WHERE gs.implementation_key = 'ACTIVE';
```

### `v_session_overview` (rename to `*_key`/`*_name`)

```sql
CREATE VIEW v_session_overview AS
SELECT es.id AS session_id,
    es.player_id,
    gt.implementation_key AS game_type_key,
    gt.name             AS game_type_name,
    gs.implementation_key AS status_key,
    cm.implementation_key AS capture_mode_key,
    es.started_at,
    es.completed_at,
    EXTRACT(EPOCH FROM (COALESCE(es.completed_at, now()) - es.started_at)) AS duration_seconds
FROM exercise_sessions es
    JOIN game_types gt    ON gt.id = es.game_type_id
    JOIN game_statuses gs ON gs.id = es.status_id
    JOIN capture_modes cm ON cm.id = es.capture_mode_id;
```

### `v_game_replay` (`*_key`; `participant` → `participant_name`)

```sql
CREATE VIEW v_game_replay AS
SELECT es.id AS session_id,
    es.player_id,
    st.sequence_number   AS stage_sequence,
    stg.implementation_key AS stage_type_key,
    t.sequence_number    AS turn_sequence,
    p.display_name       AS participant_name,
    d.dart_number,
    d.intended_target_number,
    dz1.implementation_key AS intended_zone_key,
    d.hit_target_number,
    dz2.implementation_key AS hit_zone_key,
    d.score
FROM exercise_sessions es
    JOIN exercise_stages st ON st.exercise_session_id = es.id
    JOIN stage_types stg    ON stg.id = st.stage_type_id
    JOIN turns t            ON t.exercise_stage_id = st.id
    JOIN participants p     ON p.id = t.participant_id
    JOIN darts d            ON d.turn_id = t.id
    LEFT JOIN dart_zones dz1 ON dz1.id = d.intended_zone_id
    LEFT JOIN dart_zones dz2 ON dz2.id = d.hit_zone_id;
```

### `v_dart_analytics` (`*_key`)

```sql
CREATE VIEW v_dart_analytics AS
SELECT es.player_id,
    gt.implementation_key AS game_type_key,
    d.intended_target_number,
    intended_zone.implementation_key AS intended_zone_key,
    d.hit_target_number,
    hit_zone.implementation_key AS hit_zone_key,
    d.score,
    CASE WHEN d.intended_target_number = d.hit_target_number
          AND d.intended_zone_id = d.hit_zone_id THEN TRUE ELSE FALSE END AS exact_hit
FROM darts d
    JOIN turns t             ON t.id = d.turn_id
    JOIN exercise_stages st  ON st.id = t.exercise_stage_id
    JOIN exercise_sessions es ON es.id = st.exercise_session_id
    JOIN game_types gt       ON gt.id = es.game_type_id
    LEFT JOIN dart_zones intended_zone ON intended_zone.id = d.intended_zone_id
    LEFT JOIN dart_zones hit_zone      ON hit_zone.id = d.hit_zone_id
WHERE d.intended_target_number IS NOT NULL
    AND d.intended_zone_id IS NOT NULL;
```

### `v_routine_execution` (`game_type` → `game_type_key`; `duration_type` → `duration_type_key`)

```sql
CREATE VIEW v_routine_execution AS
SELECT rt.id AS routine_id,
    rt.name AS routine_name,
    rs.sequence_number,
    et.id   AS exercise_template_id,
    et.name AS exercise_name,
    gt.implementation_key AS game_type_key,
    rs.duration_value,
    dt.implementation_key AS duration_type_key
FROM routine_templates rt
    JOIN routine_steps rs       ON rs.routine_template_id = rt.id
    JOIN exercise_templates et  ON et.id = rs.exercise_template_id
    JOIN game_types gt          ON gt.id = et.game_type_id
    JOIN duration_types dt      ON dt.id = rs.duration_type_id;
```

## 5. Non-goals (this cycle)

- **No API DTOs.** camelCase Zod response schemas, the provision-response fix, and freezing `06-API/03`+`04` are **cycle 2**.
- **No read endpoint / repository / handler implementation.** Frontend *design* does not need them yet.
- **No feature additions** beyond the `ruleset_version_key` gap. Notably, `participant_type_key` is **not** added to `v_game_replay` (it is a feature, not a normalization); revisit in a later cycle if bot/guest rendering needs it.
- **No changes to runtime/reference/template tables or seeds.** Views only.
- **No `duration_seconds` type change** — it stays `numeric` (EXTRACT epoch); DTO-level typing is a cycle-2 concern.

## 6. Key decisions

| # | Decision | Rationale |
| - | -------- | --------- |
| K1 | snake_case DB / camelCase TS (Drizzle) / camelCase JSON / Zod SoT | Standard, idiomatic per layer; already partly generated; no camelCase quoting in SQL |
| K2 | View columns: `*_key` for implementation keys, `*_name` for displayed labels only, no internal lookup ids | Consistent, key-based read contract aligned with the write side; drops leak/clutter |
| K3 | Normalize at the source (new view migration `0013`) rather than absorbing inconsistency in the repository | Robust read model; future views/queries inherit the convention; single source of truth |
| K4 | Add `ruleset_version_key` to `v_active_sessions` | Resume needs the usable key, not an internal UUID — the one functional gap |
| K5 | Behavior-preserving: rename/reproject only, same rows/joins/filters | Zero risk to replay/analytics correctness; pure read-model shape change |
